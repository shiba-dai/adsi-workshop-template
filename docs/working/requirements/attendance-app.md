# 勤怠管理 Web アプリ — 要求仕様（作業中）

## 概要

社員の出退勤を記録・管理する Web アプリケーション。

---

## ユーザーストーリー（案）

### 一般社員

| # | ストーリー |
|---|-----------|
| US-1 | 社員として、出勤打刻したい。出社時に1クリックで出勤時刻を記録できる |
| US-2 | 社員として、退勤打刻したい。退社時に1クリックで退勤時刻を記録できる |
| US-3 | 社員として、自分の勤怠履歴を一覧で確認したい。月単位で出退勤時刻と勤務時間を確認できる |
| US-4 | 社員として、打刻を修正申請したい。打刻忘れや誤りを訂正できる |

### 管理者

| # | ストーリー |
|---|-----------|
| US-5 | 管理者として、部下の勤怠一覧を確認したい。チームの出勤状況を把握できる |
| US-6 | 管理者として、打刻修正申請を承認/却下したい。不正な修正を防止できる |
| US-7 | 管理者として、月次の勤怠レポートを出力したい。集計データを CSV で取得できる |

### システム管理者

| # | ストーリー |
|---|-----------|
| US-8 | システム管理者として、社員マスタを管理したい。社員の追加・編集・無効化ができる |

## 仕様を確定するための質問

### 対象ユーザー・組織

[Question] 想定する組織規模はどの程度ですか？（例: 10名程度の小規模、50〜100名の中規模、それ以上）
[Answer]50名程度、SMB向け

[Question] ロール（権限）は「一般社員」「管理者」の2種で十分ですか？それとも「システム管理者」を分けますか？
[Answer]一般社員、管理者（管理職）、システム管理者で分ける。一般社員とライン長と、システム全体を管理する上位権限を持たせたい。

### 打刻

[Question] 打刻方法は Web 画面のボタンクリックのみですか？（他: ICカード、GPS、Slack連携 等）
[Answer]Webボタンのクリックのみ

[Question] 1日に複数回の出退勤（中抜け）を記録する必要がありますか？
[Answer]中抜けは休憩扱いにする。複数回勤怠を登録する必要はなく、1日一回のみで問題ない。

[Question] 打刻時に位置情報やIPアドレスを記録しますか？
[Answer]記録しない。

### 勤務ルール

[Question] 所定労働時間は何時間ですか？（例: 8時間）
[Answer]7時間30分

[Question] フレックスタイム制やシフト制への対応は必要ですか？（初期バージョンでは固定時間のみにするか）
[Answer]必要。所定労働時間に足りていない場合であっても受け付け不可にするのではなく、注意表示のみとする。

[Question] 休憩時間の扱いはどうしますか？（自動控除 / 手動記録 / 記録しない）
[Answer]初期で12～13時を休憩として設定、そのほかの時間についても追加で手動記録を可能にする。（必要に応じて休憩時間を追記できるようにする。）

[Question] 残業の概念は必要ですか？（所定時間超過を自動計算する等）
[Answer]必要。36協定を準拠する形が望ましい。

### 休暇・休日

[Question] 有給休暇の申請・管理機能は必要ですか？（初期スコープに含めるか）
[Answer]必要

[Question] 土日祝日の判定は必要ですか？（カレンダーマスタ等）
[Answer]必要。

### 修正・承認

[Question] 打刻修正のワークフローは「申請 → 承認」の1段階で十分ですか？
[Answer]修正は自由に実施可能とする。承認も不要。

### レポート・出力

[Question] 月次レポートに含めたい項目は？（例: 出勤日数、総労働時間、残業時間、有給消化日数）
[Answer]一般的な情報はすべて含める。

[Question] CSV 以外の出力形式（PDF 等）は必要ですか？
[Answer]CSVのみで問題ない。

### 認証・セキュリティ

[Question] 認証方式は？（ID/パスワード認証、SSO/SAML、OAuth 等）
[Answer]ID/パスワード認証

[Question] パスワードリセット機能は必要ですか？
[Answer]不要

### 技術・非機能

[Question] モバイル対応は必要ですか？（レスポンシブ Web で十分か、ネイティブアプリも必要か）
[Answer]不要

[Question] 多言語対応は必要ですか？（日本語のみで十分か）
[Answer]不要

[Question] ワークショップのスコープとして、どの機能を優先しますか？（例: まず打刻と履歴閲覧のみ実装し、修正申請やレポートは後回し）
[Answer]打刻と履歴閲覧を先に実装。そこから機能を増やしていく想定。

## セルフレビューによる追加質問

### フレックスと遅刻・早退の定義 [HIGH]

[Question] フレックスにコアタイムはありますか？（例: 10:00〜15:00）
  コアタイムがなければ「遅刻・早退」の概念が成立しません。
  コアタイムがある場合 → コアタイム違反を遅刻/早退とする。
  コアタイムがない場合 → 月次レポートから遅刻・早退の項目を削除する。
[Answer]コアタイムなし

### Phase 1 の認証 [HIGH]

[Question] Phase 1 でも最低限のログイン（認証）は必要ですか？
  打刻は「誰の」記録かを特定する必要があるため、ユーザー識別の仕組みが不可欠です。
  選択肢: (A) Phase 1 に簡易ログインを含める (B) Phase 1 は固定ユーザーで開発し、Phase 3 で認証を正式実装
[Answer]認証は必要

### 休憩自動控除の適用条件 [MEDIUM]

[Question] 休憩自動控除（12〜13時）は、その時間帯に勤務している場合のみ適用しますか？
  例: 9:00〜11:00 勤務で退勤した人には控除しない。
  例: 14:00〜22:00 勤務の人にも12〜13時控除を適用するのは不適切。
[Answer]その時間帯に勤務した人のみ、かつ6時間を超えていた場合に適用して

### 組織構造と「部下」の定義 [MEDIUM]

[Question] 管理者が確認できる「部下」はどのように決まりますか？
  (A) 部署・チーム単位で管理者を割り当てる
  (B) 社員ごとに直属の上司を設定する
  (C) その他
[Answer]A

### 有給休暇の付与ルール [MEDIUM]

[Question] 有給休暇の付与ルールは？
  - 初期付与日数（例: 入社時10日）
  - 付与タイミング（入社日基準 / 4月一斉付与）
  - 繰越上限（例: 最大20日繰越可）
  - 半日有給の有無
[Answer]
  - 初期付与日数：20日
  - 付与タイミング：4月一斉付与
  - 繰越上限：最大20日繰越可
  - 半日有給の有無：有

### 36協定上限時の動作 [MEDIUM]

[Question] 36協定上限（月45時間/年360時間）に近づいた、または超えた場合、システムはどう振る舞いますか？
  (A) 警告表示のみ（打刻は可能）
  (B) 管理者にも通知する
  (C) 打刻を拒否する
[Answer]A

## 第2回セルフレビューによる追加質問

### 月次締めと修正の関係 [MEDIUM]

[Question] 月次レポート出力後に過去の打刻を修正した場合、どう扱いますか？
  (A) 「締め」の概念を導入し、締め後は修正不可にする
  (B) いつでも修正可能。レポートは都度最新値で再集計される（過去に出力した CSV と値が変わりうる）
[Answer]A

### 部署の構造 [MEDIUM]

[Question] 部署はフラットな一覧ですか、階層構造（本部→部→課 等）ですか？
  また、1人の社員が複数部署に所属することはありますか？
[Answer]階層構造を持つ組織、一人が複数部署に所属することはあるが、メインの部署と兼務の部署等の扱いとなり、勤怠承認を行うのはメインの部署に所属する上長に限るものとする


### 管理者の兼務 [LOW]

[Question] 管理者（ライン長）自身も一般社員と同じ打刻・履歴閲覧機能を使いますか？
  （管理機能に加えて、自分自身の勤怠も管理する）
[Answer]利用する

## 第3回セルフレビューによる追加質問

### 月次締めの操作 [MEDIUM]

[Question] 月次締めは誰がどの単位で実行しますか？
  (A) 管理者が自部署単位で締める
  (B) システム管理者が全社一斉で締める
  (C) 月末から N 日後に自動で締まる
  また、締め前に社員へ「未打刻日がある」等の通知・警告は必要ですか？
[Answer]月次の承認申請を出して承諾されたタイミングを締めとする

### 管理者自身の有給承認者 [MEDIUM]

[Question] 管理者（ライン長）が有給申請をした場合、誰が承認しますか？
  (A) 上位階層の管理者（部長→本部長 等）
  (B) システム管理者が承認する
  (C) 管理者は自己承認可能（申請=承認）
[Answer]C

## 第4回セルフレビューによる追加質問

### 月次承認ワークフローの詳細 [HIGH]

月次締めが「承認申請→承諾」で行われることが判明し、新たなワークフローが必要です。

[Question] 月次勤怠の承認フローを以下の理解で合っていますか？修正があれば教えてください。
  1. 社員が「今月の勤怠を提出する」操作を行う
  2. 管理者（メイン部署の上長）が確認し、承認 or 差戻し
  3. 承認 → その月の打刻がロック（修正不可に）
  4. 差戻し → 社員が修正して再提出
[Answer]相違ない

### 月次承認が差戻しされた場合 [MEDIUM]

[Question] 差戻し時に管理者はコメント（差戻し理由）を付けられますか？
  また、差戻し回数に制限はありますか？
[Answer]コメント付与可能にしたい、差し戻し回数に制限はなし

### 管理者自身の月次承認 [MEDIUM]

[Question] 管理者自身の月次勤怠も自己承認ですか？（有給と同様）
  それとも月次勤怠は上位管理者の承認が必要ですか？
[Answer]管理者は自己承認とする
